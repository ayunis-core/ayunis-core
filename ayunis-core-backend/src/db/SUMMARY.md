Database Layer
TypeORM datasource, migrations, fixtures, and seed scripts for PostgreSQL.

The db directory manages the PostgreSQL database lifecycle. It contains the TypeORM `DataSource` initialization (with pgvector support), ~47 sequential migrations tracking every schema change, a minimal development fixture, and seed scripts that directly use TypeORM repositories to populate test data outside of NestJS dependency injection.

The `datasource.ts` file creates and exports a standalone TypeORM `DataSource` used by both the application and migration CLI. It applies a hack to register `vector` as a supported column type for pgvector compatibility. The `migrations/` directory contains chronologically ordered migration files (timestamped from `1750087277319-InitialSchema` through `1773706000224-AddLetterheadIdToArtifacts`) covering the full schema evolution: initial tables, subscriptions, models, permitted providers, legal acceptances, agents, threads, sources, RAG vectors, MCP integrations, shares, skills, teams, usage tracking, multi-provider support, and document letterhead support (`letterheads` table plus the optional `artifacts.letterheadId` foreign key). The `fixtures/` directory holds `minimal.fixture.ts` defining a canonical development dataset (a seat-based Demo Org and a usage-based Usage Org with admin users, language models across the Bedrock, Anthropic, Azure, OpenAI, Mistral, and Gemini providers, a Mistral embedding model, an Azure image-generation model, and active subscriptions with permitted models). The `scripts/seed-minimal/` directory holds the standalone minimal seed, split into focused modules: `seed-manager.ts` is both the CLI entry point and the orchestrator (the `SeedManager` class), `seed-types.ts` holds shared types, and `seed-state.ts` exposes a `SeedState` context that wraps `minimal.fixture.ts` and provides accessors (e.g. `getOrg`/`getAdmin`) shared across seeders. `seeders/` contains one idempotent seeder per concern, all extending `BaseSeeder` (which provides the shared `repo()` and `findOrCreate()` helpers): global seeders (`ModelSeeder`, `PlatformConfigSeeder`) run once, then per-org seeders (`OrgUserSeeder`, `BillingSeeder`, `UsageSeeder`, `CreditLimitSeeder`) run for each org in the fixture. Seeding is sequential — global seeders first, then each org's seeders in order. It initializes the datasource directly and uses raw TypeORM repositories (bypassing NestJS DI). The seed supports a `--clean` flag to truncate all tables before seeding. Shared `scripts/utils/` helpers provide a `SeedRunner` (initialization, password hashing, production-environment guards, `truncateAll`) and logging (`log`/`logLine`). The usage-based "Usage Org" is additionally enriched with extra users, teams, overlapping memberships, and per-user usage (so per-user and per-team credit limits can be exercised locally); that logic lives inside `usage-seeder.ts` (with a module-private `makeUsageRow` row builder), not a standalone module, and reads from the same `minimal.fixture.ts`.
