Database Layer
TypeORM datasource, migrations, fixtures, and seed scripts for PostgreSQL.

The db directory manages the PostgreSQL database lifecycle. It contains the TypeORM `DataSource` initialization (with pgvector support), ~47 sequential migrations tracking every schema change, a minimal development fixture, and seed scripts that directly use TypeORM repositories to populate test data outside of NestJS dependency injection.

The `datasource.ts` file creates and exports a standalone TypeORM `DataSource` used by both the application and migration CLI. It applies a hack to register `vector` as a supported column type for pgvector compatibility. The `migrations/` directory contains chronologically ordered migration files (timestamped from `1750087277319-InitialSchema` through `1773706000224-AddLetterheadIdToArtifacts`) covering the full schema evolution: initial tables, subscriptions, models, permitted providers, legal acceptances, agents, threads, sources, RAG vectors, MCP integrations, shares, skills, teams, usage tracking, multi-provider support, and document letterhead support (`letterheads` table plus the optional `artifacts.letterheadId` foreign key). The `fixtures/` directory holds `minimal.fixture.ts` defining a canonical development dataset (one org, admin user, Claude Sonnet 4 Bedrock language model, Text Embedding 3 Large OpenAI embedding model, active subscription with permitted models). The `scripts/seed-minimal/` package contains the minimal seed implementation: `seed-manager.ts` is the CLI entry point, `seed-state.ts` tracks produced records and exposes the loaded fixture, `seed-types.ts` defines shared types, and `seeders/` provides focused, idempotent steps (`base-seeder.ts` with a `findOrCreate` helper, plus `model-seeder.ts`, `platform-config-seeder.ts`, `org-user-seeder.ts`, `billing-seeder.ts`, `usage-seeder.ts`, `credit-limit-seeder.ts`). It initializes the datasource directly and uses raw TypeORM repositories (bypassing NestJS DI). The seed supports a `--clean` flag to truncate all tables before seeding. Shared `scripts/utils/` helpers provide `seed-runner.ts` (password hashing, production environment guards, `truncateAll`) and `seed-log.ts`. Enrichment of the "Usage Org" (extra users, teams, overlapping memberships, and per-user usage) is implemented inside `usage-seeder.ts` rather than as a standalone script; it reads from `minimal.fixture.ts` and remains idempotent.
