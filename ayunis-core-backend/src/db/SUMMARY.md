Database Layer
TypeORM datasource, migrations, fixtures, and seed scripts for PostgreSQL.

The db directory manages the PostgreSQL database lifecycle. It contains the TypeORM `DataSource` initialization (with pgvector support), ~47 sequential migrations tracking every schema change, a minimal development fixture, and seed scripts that directly use TypeORM repositories to populate test data outside of NestJS dependency injection.

The `datasource.ts` file creates and exports a standalone TypeORM `DataSource` used by both the application and migration CLI. It applies a hack to register `vector` as a supported column type for pgvector compatibility. The `migrations/` directory contains chronologically ordered migration files (timestamped from `1750087277319-InitialSchema` through `1773706000224-AddLetterheadIdToArtifacts`) covering the full schema evolution: initial tables, subscriptions, models, permitted providers, legal acceptances, agents, threads, sources, RAG vectors, MCP integrations, shares, skills, teams, usage tracking, multi-provider support, and document letterhead support (`letterheads` table plus the optional `artifacts.letterheadId` foreign key). The `fixtures/` directory holds `minimal.fixture.ts` defining a canonical development dataset (one org, admin user, Claude Sonnet 4 Bedrock language model, Text Embedding 3 Large OpenAI embedding model, active subscription with permitted models). The `scripts/seed-minimal/` directory holds the standalone minimal seed, split into focused modules: `index.ts` is the CLI entry point, `orchestrate-seed.ts` runs the seeders in dependency order, `types.ts` holds shared types, and `seeders/` contains one idempotent module per concern (organizations, models, users, subscriptions, permitted models, platform config, usage records, teams-and-usage). It initializes the datasource directly and uses raw TypeORM repositories (bypassing NestJS DI). Independent entities (org, language model, embedding model) are seeded in parallel before dependent entities (user, subscription, permitted models). The seed supports a `--clean` flag to truncate all tables before seeding. Shared `scripts/utils/` helpers provide a `SeedRunner` (password hashing, production environment guards, `truncateAll`) plus `findOrCreate`/`log`/`makeUsageRow` for the seeders. As its final step, the seed also enriches the usage-based "Usage Org" with extra users, teams, overlapping memberships, and per-user usage (so per-user and per-team credit limits can be exercised locally). That logic lives in `seeders/seed-teams-and-usage.ts`, which exports a `seedTeamsAndUsage(runner, orgId, model)` step invoked by the orchestrator rather than being a standalone entry point — it reads from the same `minimal.fixture.ts` and is idempotent.
