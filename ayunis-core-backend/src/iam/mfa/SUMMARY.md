Multi-Factor Authentication (MFA)
TOTP-based two-factor authentication with recovery codes, org-level enforcement, and admin lockout recovery.

Provides time-based one-time password (TOTP) enrollment and verification for user accounts, single-use recovery codes as a fallback, an org-level requirement policy that forces all members to enroll, and an admin reset path for locked-out users. Login-time integration is exposed to the **authentication** module through exported use cases.

Domain entities: `UserTotp` (a user's enrollment — encrypted secret, `confirmedAt` gating whether it protects login, `failedAttempts`/`lockedUntil` for lockout, and `lastUsedCounter` = the TOTP time step `floor(epoch / period)` of the last accepted code, used to reject replays), `MfaRecoveryCode` (single-use code stored as a bcrypt hash with an atomic `usedAt`), and `OrgMfaRequirement` (org policy where `required === true` forces enrollment at next login; an absent row means not required). `mfa.constants.ts` centralizes `MAX_FAILED_ATTEMPTS` (5), `LOCK_DURATION_MS` (15 min), `RECOVERY_CODE_COUNT` (10), `TOTP_WINDOW` (±1 period ≈ ±30s skew), and `TOTP_ISSUER`.

Ports (application layer): `UserTotpsRepository` (find/upsert/delete plus atomic `registerFailedAttempt` — increments and sets `lockedUntil` once the threshold is hit — and `markVerified`, which only advances `lastUsedCounter` when the new counter is strictly greater, enforcing replay protection), `MfaRecoveryCodesRepository` (transactional `replaceForUser`, atomic single-use `consume`, and count/find helpers), `OrgMfaRequirementsRepository` (find/upsert by org), `TotpSecretEncryptionPort`, and `TotpPort` (secret generation, otpauth URI + QR rendering, and `verifyCode` returning the matched absolute time step for replay tracking). `mfa.errors.ts` defines the `ApplicationError`s: `InvalidMfaCodeError` (400), `MfaLockedError` (429), `MfaAlreadyEnabledError`/`MfaRequiredByOrgError` (409), `MfaNotEnabledError` (404), and `InvalidMfaPendingTokenError`/`MfaEnrollmentNotAllowedError`/`MfaSelfResetNotAllowedError` (403 — 401 is deliberately avoided so the token-refresh filter never fires on MFA pending-state failures).

Use cases: enrollment is a two-step flow — `SetupTotpUseCase` generates a secret (abandoned unconfirmed enrollments are overwritten), stores it encrypted, and returns the secret, otpauth URI, and QR data URI; `ConfirmTotpUseCase` verifies the first code, issues the recovery codes (written before the enrollment is confirmed so a partial failure can never yield confirmed MFA without codes), and returns the plaintext codes exactly once. `VerifyMfaCodeUseCase` is the shared verification core for login and code-guarded settings operations: it accepts a TOTP code or a recovery code, counts failures in the database, and throws `MfaLockedError` after the threshold. `DisableMfaUseCase` requires a valid code and refuses when the org mandates MFA. `CheckMfaLoginRequirementUseCase` tells the login flow whether to require a code (`verify`), force enrollment (`enroll`), or proceed (`none`). `GetMfaStatusUseCase`, `GetOrgMfaRequirementUseCase`, and `UpsertOrgMfaRequirementUseCase` cover status display and org administration, and `ResetUserMfaUseCase` lets an org admin (never on themselves, org-scoped via `ContextService`) delete another user's enrollment and codes for lockout recovery.

Infrastructure: `Local*Repository` implementations back the three ports with TypeORM records (`UserTotpRecord`, `MfaRecoveryCodeRecord`, `OrgMfaRequirementRecord`) and their mappers; concurrency-sensitive operations (`registerFailedAttempt`, `markVerified`, recovery-code `consume`) run as conditional UPDATEs so races cannot bypass lockout or replay guards. `OtplibTotpAdapter` implements `TotpPort` over `otplib` (cloned with `window: TOTP_WINDOW`) and `qrcode`. `TotpSecretEncryptionService` encrypts secrets at rest with AES-256-GCM using `MFA_ENCRYPTION_KEY` (64 hex chars), output as base64(iv + ciphertext + authTag). `MfaController` exposes user settings endpoints (`GET mfa/status`, `POST mfa/setup`, `POST mfa/setup/confirm`, `DELETE mfa`) and admin endpoints (`GET`/`PUT mfa/org`, `DELETE mfa/users/:userId`, `ADMIN` role).

It integrates with **authentication** (login-time MFA via the exported `CheckMfaLoginRequirementUseCase`, `VerifyMfaCodeUseCase`, `SetupTotpUseCase`, and `ConfirmTotpUseCase`), **hashing** (`HashingModule` for bcrypt operations on recovery codes), **users** (target lookup for admin resets), and **orgs**/**users** for FK relationships.
