Budget Alerts
Budget-warning and budget-exhausted emails to org admins when a credit budget crosses a usage threshold.

This module notifies organization administrators by email when credit consumption crosses configurable thresholds (currently **50%**, **80%** and **100%**, see `application/utils/budget-alert-crossing.ts`). The 50/80 emails give admins early warning before the enforcement path (`domain/runs` guards) starts blocking runs; the 100% email announces that this is now happening. It monitors all three budget scopes that exist in the system: the org-wide monthly credit budget (from `subscriptions`), per-user credit limits and per-team credit limits (from `credit-limits`).

The work is **event-driven**: usage can only cross a threshold when tokens are consumed, so `application/listeners/budget-alerts.listener.ts` subscribes to the `TokensConsumedEvent` that `domain/usage` emits **after the usage row is persisted** — the evaluation therefore reads its own write and sees the crossing caused by that very message (the 100% email goes out with the message that exhausts the budget, before the user hits the enforcement toast). A per-org in-memory **cooldown** (10min, leading+trailing debounce) bounds how often the evaluation queries run for a busy org: the first event evaluates immediately, and usage arriving during the window triggers exactly one trailing evaluation when it expires. The trailing check deliberately needs no further event — an org that exhausts its budget inside the window gets all subsequent runs blocked, so no later event would arrive to deliver the 100% email. The cooldown is process-local (with multiple replicas each instance throttles independently) and the persisted markers make duplicate emails impossible either way; entries are removed when their window expires, so the map stays bounded. Orgs without an active usage-based subscription are skipped inside the evaluation use case (their org-wide limit resolves to `null`); events without an `orgId` (usage collected outside a request context) are ignored. Accepted gap: a budget lowered mid-month below the already-consumed amount only alerts on the org's next activity. The only scheduled work left is housekeeping: `application/tasks/budget-alert-cleanup.task.ts` (`@Cron(EVERY_DAY_AT_4AM)`) purges expired notification markers.

`EvaluateBudgetAlertsForOrgUseCase` runs inside a **per-org CLS context** (`OrgContextRunner.runForOrg`, established by the listener) because downstream lookups read the orgId from the context. `GetBudgetAlertTargetsForOrgUseCase` assembles a uniform list of `BudgetTarget`s (org + users + teams), forwarding the active subscription's `startsAt` to every usage lookup. `ProcessBudgetAlertCrossingsUseCase` loads persisted markers, delegates threshold math to the pure `application/utils/budget-alert-crossing.ts`, and handles delivery and recording.

Threshold semantics (`collectCrossings`): a target with no positive limit (unlimited org budget, or a `0` "frozen" limit) never crosses. Otherwise the percentage used is compared against each threshold. Of the thresholds that are **newly** crossed (i.e. not already recorded for this period), the **highest** is emailed and **all** are recorded — so if usage jumps straight past 80% the admin gets one 80% email, and the 50% email never fires retroactively. Each admin (resolved via `UsersModule.GetOrgAdminsUseCase`) receives a **separate** email per crossing (rendered by `SendBudgetWarningEmailUseCase` using the `BUDGET_WARNING` MJML template in `common/email-templates`). At the 100% threshold the same template renders an *exhausted* variant (subject "Budget vollständig aufgebraucht", present-tense body) instead of a percentage warning.

Idempotency is persisted through a single-table-inheritance `BudgetAlertNotificationRecord`: `scope` is the TypeORM discriminator, `orgId` identifies org targets, and the user/team child records own their respective foreign key. A database check constraint guarantees that only the ID belonging to the discriminator is populated. Unique indexes cover `(orgId, periodStart, threshold)` for org rows and `(orgId, userId/teamId, periodStart, threshold)` for user/team rows. `periodStart` is the current calendar-month start (UTC), so notifications reset each month. Only the current period is ever read; markers older than 12 months are purged by the daily cleanup cron (`CleanupBudgetAlertNotificationsUseCase`), with recent periods kept as an audit trail. Emails are sent first and the notification recorded once **at least one** admin was mailed for that crossing; all thresholds for a crossing are recorded in one database insert. A process failure between email delivery and recording can still cause a duplicate on the next run; this is deliberately at-least-once delivery. Admins whose individual send failed are deliberately not retried — that would duplicate the warning for those already mailed.

Integrations: **subscriptions** (`GetMonthlyCreditLimitUseCase` — org-wide budget), **domain/usage** (`GetMonthlyCreditUsageUseCase` — org-wide consumption, `TokensConsumedEvent` — trigger), **credit-limits** (per-user / per-team overview read models), **users** (`GetOrgAdminsUseCase` — recipients), **common/emails** + **common/email-templates** (rendering + delivery). Like `credit-limits`, this module carries the deliberate iam→domain dependency on `domain/usage`; it is a leaf module — nothing imports it.
