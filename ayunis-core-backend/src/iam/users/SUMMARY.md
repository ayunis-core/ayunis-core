User Management
Creates, validates, and manages user accounts with role-based access.

Manages the full user lifecycle including account creation (admin and regular), email confirmation, password management (reset, update, validation), profile updates, role assignment, user lookup by ID, batch of IDs (org-scoped), email, or organization, and super-admin CSV export of organization admins.

This module is the central identity store for Ayunis. The core entity is `User` with email, password hash, `UserRole` (admin/user for org-level access), `SystemRole` (customer/super-admin for platform-level access), org association, email verification status, and marketing consent. Key use cases span account creation (`CreateUserUseCase`, `CreateAdminUserUseCase`, `CreateRegularUserUseCase`), authentication support (`ValidateUserUseCase`, `IsValidPasswordUseCase`), email workflows (`ConfirmEmailUseCase`, `SendConfirmationEmailUseCase`, `ResendEmailConfirmationUseCase`), password management (`TriggerPasswordResetUseCase`, `AdminTriggerPasswordResetUseCase`, `SuperAdminTriggerPasswordResetUseCase`, `SendPasswordResetEmailUseCase`, `ResetPasswordUseCase`, `UpdatePasswordUseCase`, `ValidatePasswordResetTokenUseCase`), profile updates (`UpdateUserNameUseCase`, `UpdateUserRoleUseCase`), queries (`FindUserByIdUseCase`, `FindUsersByIdsUseCase` — batched and org-scoped via `ContextService`, used by other modules to enrich DTOs with creator/sharer display names without leaking cross-org users — `FindUserByEmailUseCase`, `FindUsersByOrgIdUseCase`, `GetOrgAdminsUseCase` — returns all org-level admins for an explicit orgId, context-free so background jobs like budget-alerts can call it without a request context), and `DeleteUserUseCase`. JWT services handle email confirmation and password reset tokens.

Super-admin reporting is served by `ExportAdminUsersUseCase`, which builds a CSV of admins belonging to organizations with a non-cancelled subscription (including subscriptions that start in the future). It reads through the `AdminUsersExportRepository` port — implemented by `LocalAdminUsersExportRepository`, a TypeORM read model that joins users, orgs, subscriptions, and teams — and is exposed over HTTP by `SuperAdminUserExportsController` (`GET super-admin/users/export/admins.csv`, `SUPER_ADMIN` only).

Domain events are defined in `application/events/`: `UserCreatedEvent` (emitted on user creation, includes optional `User` entity and department), `UserUpdatedEvent` (emitted on profile or role changes, includes updated `User` entity), `UserDeletionRequestedEvent` (emitted synchronously and awaited *before* the user row is deleted, carries userId and orgId; listeners must not act irreversibly but register cleanup via `deferCleanup` — see below), and `UserDeletedEvent` (emitted after deletion, includes userId, orgId, and email). Each event class exposes a static `EVENT_NAME` string for type-safe listener registration.

`DeleteUserUseCase` performs a compliance-grade cascade delete (AYC-382). Dependent rows — threads (and their messages, artifacts, generated-image records), knowledge bases, user system prompts, MCP user configs, user default models — are removed by `ON DELETE CASCADE` foreign keys to `users(id)` (see the `CascadeUserDeletionRelations` migration). Assets that the database cannot reach are cleaned via the two-phase contract (`DeferredCleanupEvent` in `src/common/events/`): listeners on `UserDeletionRequestedEvent` resolve cleanup targets while the rows still exist and defer the irreversible work, and `DeleteUserUseCase` runs the deferred tasks only after the row delete succeeds (each error-swallowed). A failed delete therefore loses nothing, and a failed cleanup leaks orphaned external data instead of destroying data of a still-existing user. The row delete itself (user row + pending invite) sits in its own `@Transactional` method so cleanup stays outside the transaction boundary.

It integrates with **authentication** (credential validation), **hashing** (password storage), **orgs** (org membership), **invites** (user creation on invite acceptance), and **authorization** (role definitions).
