API Keys
Org-scoped credentials for programmatic API access.

Manages creation, listing, and revocation of API keys that authenticate external programmatic access to Ayunis. Keys are scoped to an organization (not a user) and managed by org admins. The full plaintext key is shown only once at creation time; the database stores a bcrypt hash plus an indexed plaintext lookup prefix for fast lookup at request time.

Key entity is `ApiKey` (id, name, prefix, hash, expiresAt, revokedAt, orgId, createdByUserId, timestamps), exposing `KEY_PREFIX = 'ayk_live_'` and `LOOKUP_PREFIX_LENGTH = 12` as static constants so the use case, the DTO mapper, and the request-time verifier all share one source of truth for the key format. Use cases include `CreateApiKeyUseCase` (generates a 256-bit random secret formatted as `ayk_live_<base64url>`, derives the lookup prefix, hashes the full secret via `HashTextUseCase`, persists with one retry on the rare prefix-collision case), `ListApiKeysByOrgUseCase` (returns keys for the caller's org, including revoked keys for audit-trail visibility), `RevokeApiKeyUseCase` (soft-deletes a key via a conditional `UPDATE … SET revoked_at = NOW() WHERE id = $1 AND revoked_at IS NULL` so concurrent revokes preserve the original timestamp; surfaces `ApiKeyNotFoundError` for both truly-missing and cross-org cases so callers cannot enumerate IDs across organizations — cross-org attempts are still logged at warn level for incident review), and `ValidateApiKeyUseCase` (the request-time verifier: parses the lookup prefix from the presented token, fetches the candidate row via `findByPrefix`, and authenticates with `CompareHashUseCase`; unknown prefix, malformed token, hash mismatch, and revoked-key cases all collapse to a single `ApiKeyNotFoundError` so callers cannot distinguish failure modes, while expired keys raise `ApiKeyExpiredError` separately — the `apiKeyId` is logged at warn level for forensics but deliberately omitted from the public payload per AYC-77 S4). Authorization is enforced at the controller via `@Roles(UserRole.ADMIN)`; the create route additionally requires `@RequireSubscription({ type: SubscriptionType.USAGE_BASED })` so only orgs with a usage-based subscription (or remaining trial messages) can mint new keys, and create/revoke routes are rate-limited via `@RateLimit`. Integrates with **hashing** for bcrypt operations and **orgs**/**users** for FK relationships (cascade-delete on org removal, set-null on creator removal to preserve audit).

For data-privacy compliance, `PurgeExpiredApiKeysUseCase` (driven by the nightly `PurgeExpiredApiKeysTask` cron at 5 AM) hard-deletes keys whose `expiresAt` is older than a configurable grace period (`PURGE_API_KEY_GRACE_DAYS`, default 30 days) so expired credentials are not retained indefinitely. Keys without an expiry — including revoked-but-not-expired keys kept for audit — are never touched (a `expires_at < cutoff` predicate never matches `NULL`). A `PURGE_DRY_RUN` flag makes the job report the impact without deleting.

The matching `ApiKeyStrategy` lives in the **authentication** module. The OpenAI-compatible chat completions endpoint that consumes `ValidateApiKeyUseCase` follows in iter 3 of AYC-92.
