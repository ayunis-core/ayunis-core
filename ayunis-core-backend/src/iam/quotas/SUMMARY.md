Usage Quotas
Tracks and enforces per-user usage limits with sliding windows.

Implements a quota enforcement system that tracks user resource consumption within configurable time windows. Currently supports tier-aware fair-use message limits and a separate fair-use image-generation limit, preventing abuse while allowing reasonable platform usage.

This module provides rate-based usage control for Ayunis. The core entity is `UsageQuota`, which tracks a user's consumption count for a specific `QuotaType` (currently `FAIR_USE_MESSAGES_LOW`, `FAIR_USE_MESSAGES_MEDIUM`, `FAIR_USE_MESSAGES_HIGH` — one independent counter per model tier — and `FAIR_USE_IMAGES`, an independent counter for image-generation calls) within a sliding time window defined by `windowStartAt` and `windowDurationMs`. The entity includes business logic methods like `isWindowExpired()` to detect when a window has elapsed. Key components are `CheckQuotaUseCase` for verifying whether a user has remaining quota before allowing an operation, and `QuotaLimitResolverService` for determining the applicable limit and window for a given quota type. The resolver reads live tiered limits from `PlatformConfigModule.GetFairUseLimitsUseCase` on every call so super-admin platform-config updates take effect immediately (no restart, no caching); it routes message quota types to the matching tier bucket and `FAIR_USE_IMAGES` to the dedicated `images` bucket. The `domain/tier-to-quota-type.ts` helper maps a `ModelTier` to its message bucket; untiered models default to MEDIUM, and the `ZERO` tier returns `null` to signal that the model is exempt from fair-use enforcement (no bucket, no row in `usage_quotas`). Note that the platform-config layer still persists `FAIR_USE_ZERO_*` rows for UI symmetry — the super admin can configure a value, but the resolver never reads it because ZERO short-circuits before reaching `QuotaLimitResolverService.resolve()`. Image enforcement is independent of the tier mapping. `QuotaExceededError` messages intentionally omit super-admin-configurable details (quota type, limit, window) so they don't leak through logs, HTTP response bodies, or LLM tool results; consumers that need those values read them from the error's `metadata`. It integrates with **platform-config** for live limit/window resolution, **users** for user identification, and the **domain/runs** module which checks quotas inside `ExecuteRunUseCase` (after the model is resolved) before executing AI model calls. The `UsageQuotaRepositoryPort` abstracts persistence with a PostgreSQL implementation that supports atomic increment operations.
