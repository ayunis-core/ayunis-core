Organization Management
Creates, updates, deletes, and queries multi-tenant organization entities.

Manages organizations as the top-level multi-tenancy boundary in Ayunis. Supports CRUD operations, super-admin listing of all organizations, and lookup by ID. Each user belongs to exactly one organization.

This module defines the `Org` entity—the fundamental tenant unit containing a name, associated users, and timestamps. Key use cases include `CreateOrgUseCase` (provisions a new organization, typically during registration), `UpdateOrgUseCase` (renames an organization), `DeleteOrgUseCase` (removes an organization and cascades cleanup), `FindOrgByIdUseCase` (single org lookup), `FindAllOrgIdsUseCase` (batch ID retrieval), and `SuperAdminGetAllOrgsUseCase` (paginated listing for platform administration). The HTTP layer includes a `SuperAdminOrgsController` for admin-only endpoints.

Domain events are defined in `application/events/`: `OrgCreatedEvent` (emitted by `CreateOrgUseCase` whenever a new organization is created — during registration or via super-admin — and includes the `Org` entity). The event class exposes a static `EVENT_NAME` string for type-safe listener registration.

It integrates with **users** (each user references an orgId), **subscriptions** and **trials** (org-scoped billing), **invites** (org-scoped invitations), **teams** (org-scoped team grouping), and **authentication** (org creation during registration). The `OrgsRepositoryPort` abstracts persistence with a PostgreSQL implementation.

**Deletion cascade contract.** `DeleteOrgUseCase` issues a single `DELETE` on the `orgs` row and relies entirely on PostgreSQL `ON DELETE CASCADE` foreign keys for cleanup — it does not orchestrate per-module deletions. Every org-scoped table therefore MUST declare a `@ManyToOne(() => OrgRecord, { onDelete: 'CASCADE' })` (directly, or transitively through a parent that cascades from the org). Conversation data cascades through `users` (`threads.userId → users`, then messages/artifacts/assignments via `threadId`); knowledge-base data cascades through `knowledge_bases.orgId → orgs`, then `sources.knowledgeBaseId → knowledge_bases` and the RAG parent/child chunks below sources; MCP data cascades through `mcp_integrations.orgId → orgs`. When adding a new org-scoped table, wire its cascade here — a plain `orgId`/`userId` column without an FK silently orphans data on org deletion (a data-protection defect). Note that cascades cover relational rows and pgvector embeddings (all in Postgres) but NOT external blob storage (uploaded files, generated images); those are cleaned by the per-entity delete use cases, not by org deletion.
