Access Control
Guards and decorators enforcing role-based access control rules.

Provides authorization infrastructure through NestJS guards and decorators for role-based access control, subscription validation, email confirmation checks, and rate limiting across all API endpoints.

This module supplies the cross-cutting authorization layer for Ayunis. It contains no domain entities of its own but defines guards and decorators consumed by every controller. Key components include `RolesGuard` with `@Roles()` decorator for org-level admin restrictions (checks `UserRole.ADMIN` or `USER`), `SystemRolesGuard` with `@SystemRoles()` for platform-level system role checks, `SubscriptionGuard` with `@RequireSubscription()` to gate features behind active subscriptions (the decorator accepts an optional `{ type?: SubscriptionType }` filter that restricts the subscription branch of the gate to a specific subscription type — e.g. `USAGE_BASED` — while the trial fallback still applies regardless), `EmailConfirmGuard` to restrict unverified users, and `RateLimitGuard` with `@RateLimit()` for request throttling. `SubscriptionGuard` is bound globally via `APP_GUARD` (using `useExisting` so it's a singleton) AND re-exported as a provider so other modules (e.g. `OpenAICompatModule`) can bind it locally at controller level — necessary on `@Public()` routes where the global instance defers because no principal is on the request yet and a controller-level auth guard like `AuthGuard('api-key')` still needs to populate `request.user`. The guard resolves both `ActiveUser` and `ApiKeyPrincipal` from `request.user` so it works for both browser-session and API-key-authenticated routes. It depends on **authentication** for the `ActiveUser` entity attached to requests and **users** for role value objects. Authorization errors are defined locally and converted to HTTP 403 responses via `toHttpException()`. When `RolesGuard` or `SystemRolesGuard` reject a request, they emit a structured `Logger.warn` audit entry (via `buildAccessDeniedAuditContext` in `common/util`) capturing the principal, required vs. actual role, and request method/path/IP, so 403s caused by insufficient role are no longer silent.

The guards are declared as plain providers here and exported for consumers; this module does **not** wire the global `APP_GUARD` bindings itself. `IamModule` owns those bindings and their execution order (so the order between authentication and authorization guards is declared in one place rather than derived from NestJS module-scan order).
