Credit Limits
Configurable per-user and per-team monthly credit allowances, enforced on top of the org-wide credit budget.

This module lets organization admins cap the monthly credit consumption of individual users and of teams, independently of (and in addition to) the org-wide credit budget enforced by the subscriptions/runs `CreditBudgetGuardService` and the fair-use quotas. It exists because orgs on usage-based subscriptions previously had no way to stop a single user or team from depleting the entire monthly credit allowance.

The core entity is `CreditLimit` (`domain/credit-limit.entity.ts`): an org-scoped row carrying a `scope` (`USER` | `TEAM`), exactly one target (`targetUserId` xor `targetTeamId`, DB-enforced by the `CHK_credit_limit_target_xor` check constraint), and a `monthlyCredits` allowance. Construct via the class constructor using a `CreditLimitTarget` discriminated union, which guarantees the scope↔target pairing. Semantics: the **presence** of a row means a cap is set; the **absence** means the principal is unlimited within the org budget; a value of **0** freezes the principal entirely. Unit is credits — the same unit as `UsageBasedSubscription.monthlyCredits` and `usage.creditsConsumed`. Persistence uses unique partial indexes (`UQ_credit_limit_user`, `UQ_credit_limit_team`) so each user/team has at most one limit; both target FKs cascade-delete so removing a user/team removes their limit.

Write use cases (`SetUserCreditLimitUseCase`, `SetTeamCreditLimitUseCase`) upsert by (orgId, target), preserving row identity so a limit can be raised/lowered idempotently; they reject negative `monthlyCredits`. `RemoveCreditLimitUseCase` deletes a limit (idempotent). All write/list use cases read `orgId` from `ContextService` and are exposed only to `UserRole.ADMIN` via `CreditLimitsController` (`GET /credit-limits`, `PUT|DELETE /credit-limits/users/:userId`, `PUT|DELETE /credit-limits/teams/:teamId`).

Two read paths exist. `GetCreditLimitsForUserUseCase` is the **enforcement** read model: given an explicit principal (orgId + userId, NOT context — it is called by the runs guard), it returns the user's personal limit plus the limits of every team they belong to (resolved via `TeamsModule.FindTeamsByUserIdUseCase`). `GetCreditLimitsOverviewUseCase` is the **admin UI** read model: it enriches every configured limit with the target's display name (`UsersModule.FindUsersByIdsUseCase`, `TeamsModule.ListTeamsUseCase`) and current monthly consumption (`UsageModule.GetMonthlyCreditUsageForUsers` — one batched per-user query covering all configured user limits — and `GetMonthlyCreditUsageForTeam` per team).

Enforcement itself lives in the runs module (`CreditLimitGuardService`), not here, because it combines this module's config with usage measurement and team membership — a cross-domain decision the runs flow owns. That guard is invoked from `InferenceUsageGuard.preflight()` after the org credit-budget check, only for user principals (api-key principals skip, matching fair-use), and applies most-restrictive-wins: a run is blocked if the user's personal limit OR any of their teams is exhausted. A team limit is a shared pool — the sum of every current member's consumption — measured over the current calendar month (UTC).

Integrations: **teams** (membership + names), **users** (names), **domain/usage** (consumption measurement), and **domain/runs** (the enforcement guard that consumes `GetCreditLimitsForUserUseCase`). This module is feature-relevant only for usage-based subscriptions; the frontend hides the settings section for seat-based orgs.

> **Deliberate cross-context dependency (iam → domain):** `GetCreditLimitsOverviewUseCase` injects `domain/usage` use cases (`GetMonthlyCreditUsageForUsers/Team`) to show live consumption next to each configured limit, so `CreditLimitsModule` imports `UsageModule`. This is the only iam→domain *module* dependency in the codebase — the established direction is domain→iam. It is intentional and acyclic (`UsageModule` does not depend back on this module): an admin endpoint that displays consumption inherently needs domain usage data, and per the cross-module convention we inject the exported use case directly rather than wrapping it in a port. The enforcement path keeps the conventional direction — `CreditLimitGuardService` lives in `domain/runs` and depends on this iam module, not the reverse.
