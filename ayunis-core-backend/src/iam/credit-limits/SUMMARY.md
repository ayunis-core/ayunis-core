Credit Limits
Configurable per-user and per-team monthly credit allowances, enforced on top of the org-wide credit budget.

This module lets organization admins cap the monthly credit consumption of individual users and of teams, independently of (and in addition to) the org-wide credit budget enforced by the subscriptions/runs `CreditBudgetGuardService` and the fair-use quotas. It exists because orgs on usage-based subscriptions previously had no way to stop a single user or team from depleting the entire monthly credit allowance.

The core entity is `CreditLimit` (`domain/credit-limit.entity.ts`): an abstract, org-scoped base carrying a `monthlyCredits` allowance, with one concrete subclass per scope — `UserCreditLimit` (owns `userId`) and `TeamCreditLimit` (owns `teamId`). The subclass identity *is* the discriminant, so no `scope` field is carried on the domain entity; the `CreditLimitScope` enum lives only where a value is genuinely needed — the STI `@ChildEntity` discriminator and the `scope` field of the HTTP response DTO. Modeling the target as a subclass (rather than a union field) mirrors the `Tool` / `MessageContent` domain convention and the record layer 1:1, and makes an invalid "both / neither target" state unrepresentable. Construct the subtype directly — `new UserCreditLimit({ orgId, userId, monthlyCredits })` / `new TeamCreditLimit({ orgId, teamId, monthlyCredits })`. Because every endpoint (and therefore every read) is scope-specific, the repository and use cases are typed to the concrete subtype end-to-end (`findUserLimits` → `UserCreditLimit[]`, `SetTeamCreditLimitUseCase` → `TeamCreditLimit`, etc.); the abstract `CreditLimit` base survives only at the polymorphic write seam — `save<T extends CreditLimit>` and the persistence mapper, which `instanceof`-dispatches a domain subtype to its record subtype. Persistence uses TypeORM single-table inheritance (`CreditLimitRecord` → `UserCreditLimitRecord` / `TeamCreditLimitRecord`, discriminated by the `scope` column); each subtype owns its target FK, and partial unique indexes (on `userId` where non-null, on `teamId` where non-null) give each user/team at most one limit. Both target FKs cascade-delete, so removing a user/team removes their limit. Semantics: the **presence** of a row means a cap is set; the **absence** means the principal is unlimited within the org budget; a value of **0** freezes the principal entirely. Unit is credits — the same unit as `UsageBasedSubscription.monthlyCredits` and `usage.creditsConsumed`.

Write use cases (`SetUserCreditLimitUseCase`, `SetTeamCreditLimitUseCase`) upsert by (orgId, userId/teamId), preserving row identity so a limit can be raised/lowered idempotently; they reject negative `monthlyCredits`. `RemoveUserCreditLimitUseCase` / `RemoveTeamCreditLimitUseCase` delete a limit (idempotent). All write and admin-overview use cases read `orgId` from `ContextService` and are exposed only to `UserRole.ADMIN` via `CreditLimitsController` (`GET /credit-limits/users`, `GET /credit-limits/teams`, `PUT|DELETE /credit-limits/users/:userId`, `PUT|DELETE /credit-limits/teams/:teamId`).

Two read paths exist. `ResolveCreditLimitsForUserUseCase` is the **enforcement** read model: given an explicit principal (orgId + userId, NOT context — it is called by the runs guard), it returns the user's personal limit plus the limits of every team they belong to (resolved via `TeamsModule.FindTeamsByUserIdUseCase`). `GetUserCreditLimitsOverviewUseCase` and `GetTeamCreditLimitsOverviewUseCase` are the **admin UI** read models — one per scope, so each settings page fetches only its own list rather than a combined overview. Each enriches its configured limits with the target's display name (`UsersModule.FindUsersByIdsUseCase` / `TeamsModule.ListTeamsUseCase`) and current monthly consumption (`UsageModule.GetMonthlyCreditUsageForUsers` — one batched per-user query — and `GetMonthlyCreditUsageForTeam` per team). Both overview use cases accept an optional `since` lower bound for the usage window and are also consumed by the `budget-alerts` module's alert evaluation (the HTTP controller calls them without a query, so `since` is not reachable from the API).

Enforcement itself lives in the runs module (`CreditLimitGuardService`), not here, because it combines this module's config with usage measurement and team membership — a cross-domain decision the runs flow owns. That guard is invoked from `InferenceUsageGuard.preflight()` after the org credit-budget check, only for user principals (api-key principals skip, matching fair-use), and applies most-restrictive-wins: a run is blocked if the user's personal limit OR any of their teams is exhausted. A team limit is a shared pool — the sum of every current member's consumption — measured over the current calendar month (UTC).

Integrations: **teams** (membership + names), **users** (names), **domain/usage** (consumption measurement), and **domain/runs** (the enforcement guard that consumes `ResolveCreditLimitsForUserUseCase`). This module is feature-relevant only for usage-based subscriptions; the frontend hides the settings section for seat-based orgs.

> **Deliberate cross-context dependency (iam → domain):** `GetUserCreditLimitsOverviewUseCase` / `GetTeamCreditLimitsOverviewUseCase` inject `domain/usage` use cases (`GetMonthlyCreditUsageForUsers/Team`) to show live consumption next to each configured limit, so `CreditLimitsModule` imports `UsageModule`. Together with `budget-alerts` (which follows the same pattern), these are the only iam→domain *module* dependencies in the codebase — the established direction is domain→iam. It is intentional and acyclic (`UsageModule` does not depend back on this module): an admin endpoint that displays consumption inherently needs domain usage data, and per the cross-module convention we inject the exported use case directly rather than wrapping it in a port. The enforcement path keeps the conventional direction — `CreditLimitGuardService` lives in `domain/runs` and depends on this iam module, not the reverse.
