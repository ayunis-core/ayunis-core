Credit Alerts
Daily budget-warning emails to org admins when a credit budget crosses a usage threshold.

This module notifies organization administrators by email when credit consumption crosses configurable thresholds (currently **50%** and **80%**, see `application/credit-alerts.constants.ts`). It complements the enforcement path (`domain/runs` guards block runs once a budget is *exhausted*) by giving admins early warning *before* that happens. It monitors all three budget scopes that exist in the system: the org-wide monthly credit budget (from `subscriptions`), per-user credit limits and per-team credit limits (from `credit-limits`).

The work is driven by a daily cron (`infrastructure/tasks/budget-alerts.task.ts`, `@Cron(EVERY_DAY_AT_4AM)`, early-morning UTC to match the other cleanup jobs, guarded by an `isRunning` flag). It lists every org via `OrgsModule.FindAllOrgIdsUseCase` and calls `EvaluateBudgetAlertsForOrgUseCase` per org; a per-org failure is logged and never aborts the sweep. Orgs without an active usage-based subscription are skipped inside the evaluation use case (their org-wide limit resolves to `null`).

`EvaluateBudgetAlertsForOrgUseCase` runs inside a **per-org CLS context** (`contextService.run` + `set('orgId', …)`, the same pattern as `domain/retention-policies` enforcement) so it can reuse the org-scoped, context-reading overview use cases from `credit-limits` (`GetUserCreditLimitsOverviewUseCase` / `GetTeamCreditLimitsOverviewUseCase`, which already return each limit alongside its live consumption) without duplicating enrichment. It assembles a uniform list of `BudgetTarget`s (org + users + teams), then delegates the threshold math to the pure `application/budget-alert-crossing.ts`.

Threshold semantics (`collectCrossings`): a target with no positive limit (unlimited org budget, or a `0` "frozen" limit) never crosses. Otherwise the percentage used is compared against each threshold. Of the thresholds that are **newly** crossed (i.e. not already recorded for this period), the **highest** is emailed and **all** are recorded — so if usage jumps straight past 80% the admin gets one 80% email, and the 50% email never fires retroactively. Each admin (resolved via `UsersModule.GetOrgAdminsUseCase`) receives a **separate** email per crossing (rendered by `SendBudgetWarningEmailUseCase` using the `BUDGET_WARNING` MJML template in `common/email-templates`).

Idempotency is persisted: `BudgetAlertNotification` (one row per `orgId + scope + targetId + periodStart + threshold`, unique index enforced) marks "already notified". `targetId` is the org id for ORG scope and the user/team id otherwise, so it is always non-null and backs the unique index directly (it is polymorphic, hence carries no FK — only `orgId` is a real foreign key, cascade-deleting with the org). `periodStart` is the current calendar-month start (UTC), so notifications reset each month. Emails are sent first and the notification recorded only after all of an org's admins were mailed for that crossing; if sending throws, the crossing is left unrecorded and retried on the next daily run.

Integrations: **orgs** (org iteration), **subscriptions** (`GetMonthlyCreditLimitUseCase` — org-wide budget), **domain/usage** (`GetMonthlyCreditUsageUseCase` — org-wide consumption), **credit-limits** (per-user / per-team overview read models), **users** (`GetOrgAdminsUseCase` — recipients), **common/emails** + **common/email-templates** (rendering + delivery). Like `credit-limits`, this module carries the deliberate iam→domain dependency on `domain/usage`; it is a leaf module — nothing imports it.
