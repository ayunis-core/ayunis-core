Configuration Registry
Centralized environment-based configuration factories for all backend services and integrations.

The config directory contains two categories of files:

**Environment Bootstrap** â€” `env.ts` and `env-register.js` handle dotenv loading before any other code runs. `env.ts` is imported as the first line in TypeScript entry points (`main.ts`, `cli/main.ts`, seed scripts). `env-register.js` is a CommonJS preload script for `-r` flag usage with ts-node/TypeORM CLI. Both load `.env.dev` (per-slot overrides from `./dev`) then `.env` (defaults), ensuring dev environment variables take precedence.

**NestJS registerAs Factories** â€” The `*.config.ts` files are NestJS `registerAs` configuration factories that map environment variables to typed configuration objects. Each file covers a specific concern: admin tokens, app settings, authentication/JWT, emails/SMTP, embeddings, evals/Langfuse, feature toggles, internet search/Brave, legal versioning, Locaboo, MCP URLs, model provider API keys, retrieval/Gotenberg, storage/MinIO, subscriptions/pricing, tools limits, TypeORM/Postgres, URL retriever limits, and web/CORS.

The configuration module provides strongly-typed, namespaced configuration for every backend subsystem via NestJS `registerAs` factories. `app.config.ts` controls port, deployment mode (cloud/self-hosted), registration toggle, frontend URLs, webhook endpoints (`ORG_EVENTS_WEBHOOK_URL`), and the HMAC-SHA256 outbound webhook signing secret (`WEBHOOK_SIGNING_SECRET`). `authentication.config.ts` configures JWT secrets, token expiration for access/refresh/invite/email-confirmation/password-reset tokens, cookie settings, and auth provider selection (local vs cloud). `models.config.ts` holds API keys and base URLs for all LLM providers (Mistral, OpenAI, Anthropic, Bedrock, Ollama, Synaforce, Ayunis, OTC, Gemini, Azure). `typeorm.config.ts` builds PostgreSQL `DataSourceOptions` with environment-specific entity/migration paths and SSL settings for production. `storage.config.ts` defines MinIO connection parameters. `subscriptions.config.ts` sets per-seat pricing and trial message limits. `emails.config.ts` maps SMTP host/port/credentials. `legal.config.ts` tracks versioned terms-of-service and provider-specific legal acceptances. `features.config.ts` defines feature toggles (`FeatureFlag` enum) for agents, knowledge bases, prompts, and skills, each controlled by a `FEATURE_*_ENABLED` environment variable with sensible defaults. `url.config.ts` (the `url` namespace) bounds the URL retriever's outbound fetches: `URL_RETRIEVER_TIMEOUT_MS` (fetch timeout, default 5s) and `URL_RETRIEVER_MAX_DOWNLOAD_BYTES` (maximum bytes buffered from a fetched URL, default 25 MB). The root-level factories are collected in `root-configs.ts` (`rootConfigs`), which every Nest entrypoint (`AppModule`, `CliModule`) passes to `ConfigModule.forRoot({ load: rootConfigs })` â€” entrypoints must load the shared list rather than hand-picking namespaces, because their transitive module graphs read namespaced keys (e.g. `auth.jwt.secret` via `getOrThrow`) that fail at bootstrap if the namespace is missing. `authentication.config.ts` requires `JWT_SECRET` and `COOKIE_SECRET` in every environment and throws at load time when either is missing; there is no insecure default fallback. Feature-scoped factories (e.g. `retrieval.config.ts`) are loaded by their own modules via `ConfigModule.forFeature`.
