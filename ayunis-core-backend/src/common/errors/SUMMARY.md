Error Foundation
Base application error class and unauthorized access error definition.

The errors directory defines the `ApplicationError` abstract base class that all domain-specific errors extend. It provides error code, HTTP status code, metadata, and a `toHttpException()` method that maps to NestJS exceptions. The `UnauthorizedAccessError` is a concrete 403 error for authorization failures.

The `ApplicationError` base class is the cornerstone of the backend's error handling strategy. Every domain module defines its own error hierarchy extending this class (e.g., `AgentError`, `ThreadError`). Each error carries a string `code` for programmatic identification, a numeric `statusCode` for HTTP mapping, and optional `metadata` for additional context. The `toHttpException()` method converts domain errors to NestJS HTTP exceptions (`NotFoundException`, `BadRequestException`, `ConflictException`, `ForbiddenException`) based on status code, preserving code and metadata in the response body. The companion `ApplicationErrorFilter` in `../filters/` catches these errors globally, reports unexpected (5xx) errors to AppSignal, and returns structured JSON error responses with timestamps and request paths. The `UnauthorizedAccessError` is a reusable 403 error used across modules when repository-level authorization checks fail.

The directory also hosts the shared provider-failure taxonomy for outbound adapters (LLM inference, embeddings, OCR):

- `provider.errors.ts` — the `ProviderUnavailableError` family (`ProviderConnectionError` 502, `ProviderTimeoutError` 504, `ProviderServerError` 502, `ProviderRequestRejectedError` 502). Each instance sets `name === code === PROVIDER_UNAVAILABLE_<CLASS>_<PROVIDER>` so AppSignal groups one incident per provider+failure-class on both reporting paths (HTTP `setError` groups by `name`; the BullMQ OTel `recordException` path prefers `code`). Metadata carries provider, model, host, underlying errno, and upstream status.
- `provider-transport-error.classifier.ts` — `classifyTransportError()` detects connection/timeout failures by walking an error's `cause` chain (undici `fetch failed` wrappers, `AggregateError` multi-address connects), matching errno/undici/TLS codes and SDK wrapper names.
- `extract-upstream-status.helper.ts` — normalizes the upstream HTTP status from provider SDK error shapes (OpenAI/Anthropic `status`, legacy `statusCode`, fetch `response.status`, AWS `$metadata.httpStatusCode`).
- `wrap-provider-failure.helper.ts` — `wrapProviderFailure()`, the single integration API for adapters/use-cases: returns a `ProviderUnavailableError` for transport failures and upstream 5xx/408, `undefined` for `ApplicationError`, upstream 4xx (kept distinct as potentially-our-bug), and unrecognized errors.
