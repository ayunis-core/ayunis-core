Request Context
Provide request-scoped principal identity via continuation-local storage service.

The context module wraps `nestjs-cls` to expose a `ContextService` that stores and retrieves the current principal's identity throughout a request lifecycle. It is globally available and used by all use cases to access authentication context without parameter passing.

The context module is a global wrapper around the `nestjs-cls` `ClsService`, typed with the `MyClsStore` interface containing `principalKind`, `userId`, `apiKeyId`, `orgId`, `systemRole`, and `role` fields. The `ContextModule` is decorated `@Global()` and provides `ContextService` as a standalone `@Injectable()` class that wraps `ClsService` with domain-specific helpers. Authentication middleware populates the store early in the request pipeline after JWT validation or API-key resolution. The `principalKind` field discriminates between `'user'` and `'apiKey'` principals — when the principal is a user, `userId` is set; when it is an API key, `apiKeyId` is set. `ContextService` exposes `get()`, `set()`, and `run()` for raw store access, plus two convenience methods: `requireOrgId()` returns the current org ID or throws `UnauthorizedAccessError`, and `requirePrincipal()` returns a discriminated `ResolvedPrincipal` union (user or apiKey variant) or throws `UnauthorizedAccessError`. Use cases should prefer these helpers over manual `get()` + null checks. The store also carries `systemRole` for super-admin checks and `role` for organization-level authorization decisions.
