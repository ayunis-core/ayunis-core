Extensible Tools
Pluggable tool system with JSON schema validation and execution

Tools provide an extensible capability system where AI agents invoke typed actions—HTTP calls, code execution, chart generation, image generation, source queries, internet search, email, calendar, document editing/export preparation, and MCP operations—during conversations.

The tools module implements a polymorphic tool system with JSON schema-based parameter validation. The abstract `Tool` entity specializes into 15+ concrete types: `HttpTool`, `CodeExecutionTool`, `InternetSearchTool`, `WebsiteContentTool`, `SourceQueryTool`, `SourceGetTextTool`, `KnowledgeQueryTool`, `KnowledgeGetTextTool`, `SendEmailTool`, `CreateCalendarEventTool`, chart tools (bar, line, pie), `McpIntegrationTool`, `McpIntegrationResource`, `McpIntegrationPrompt`, `ProductKnowledgeTool`, `ActivateSkillTool`, `CreateDocumentTool`, `UpdateDocumentTool`, `EditDocumentTool`, and `GenerateImageTool`. `ToolBox` groups tools by type, and `DisplayableTool`/`ConfigurableTool` add UI and configuration capabilities. Each tool type has a dedicated handler for execution — `KnowledgeQueryToolHandler` performs semantic search across a knowledge base's documents, and `KnowledgeGetTextToolHandler` retrieves raw document text by line range from a knowledge base document. The `ActivateSkillToolHandler` routes skill activations by prefix: `user__` slugs activate user skills via `SkillActivationService` (injecting sources and MCP integrations into the thread), while `system__` slugs look up always-on skill templates via `FindAlwaysOnTemplateByNameUseCase` and return the template instructions directly. The `ActivateSkillTool` entity carries a `Map<slug, originalName>` to resolve slugs back to original names without lossy de-slugification. The document tools (`CreateDocumentTool`, `UpdateDocumentTool`, `EditDocumentTool`) manage versioned artifacts during conversations — `CreateDocumentToolHandler` creates new artifacts, `UpdateDocumentToolHandler` replaces content wholesale or changes the artifact's letterhead assignment, and `EditDocumentToolHandler` applies surgical search-and-replace edits. `CreateDocumentTool` and `UpdateDocumentTool` both accept an optional `letterhead_id` so the model can attach an org-managed letterhead when the user requests an official document. `GenerateImageTool` (extends `Tool`, not `DisplayableTool`) accepts a prompt-only parameter and is backend-only executable; `GenerateImageToolHandler` resolves the org's permitted image model via `GetPermittedImageGenerationModelUseCase` (models module), generates the image via Azure using `GenerateImageUseCase` (models module), saves it to storage via `SaveGeneratedImageUseCase` (threads module), and returns the generated image's stable UUID as the bare tool result. `ToolAssemblyService` conditionally includes the tool only when the org has a permitted image generation model. Shared utility modules support these handlers: `embedding-error.utils.ts` provides consistent error handling for embedding failures, and `text-extraction.utils.ts` implements line-range text extraction with validation and clamping logic. Key use cases include assembling tools for agents, executing tool calls, and checking tool capabilities. The module integrates with **agents** for tool assignment, **runs** for runtime tool execution, **sources** and **rag** for knowledge retrieval tools, **knowledge-bases** for knowledge base scoped retrieval tools, **mcp** for protocol-bridged tools, **retrievers** for search and content tools, **artifacts** for document persistence, **skill-templates** for always-on template lookup during skill activation, **models** for permitted image generation model resolution and image generation, and **threads** for generated image storage.
