Conversation Sessions
Conversation threads linking messages, models, and sources

Threads are conversation containers that hold message histories, bind to a language model and optional agent, and support source attachments and knowledge base assignments for per-conversation RAG context enrichment.

The threads module manages conversation sessions. The `Thread` entity holds a message list, optional `PermittedLanguageModel` reference, optional agent binding, source assignments, knowledge base assignments, title, and anonymous flag. `SourceAssignment` links data sources to individual threads and carries an optional `originSkillId` provenance field that tracks which skill caused the source to be attached — this enables cleanup when a skill share is revoked. The `KnowledgeBaseAssignment` entity represents a knowledge base attached to a thread; the underlying `thread_knowledge_base_assignments` table stores the thread ID, knowledge base ID, and an optional `originSkillId` provenance column that tracks which skill caused the assignment. This provenance tracking enables targeted cleanup when a skill share is revoked or when a knowledge base is removed from a skill — only assignments originating from that skill are deleted, leaving user-initiated (direct) assignments intact. Key use cases include creating threads, finding threads (single, all, by org with sources), updating titles, deleting threads (idempotent — treats already-deleted or not-found threads as success), adding/removing sources, adding uploaded file sources (`AddFileSourceToThreadUseCase` — detects the file type, dispatches documents to processing and CSV/spreadsheet files to data-source creation via the sources module's `DataSourceCommandBuilderService`, and attaches every created source to the thread; the CSV and spreadsheet paths are transactional), adding/removing knowledge bases from threads (`AddKnowledgeBaseToThreadUseCase`, `RemoveKnowledgeBaseFromThreadUseCase`), removing skill-originated KB assignments (`RemoveKnowledgeBaseAssignmentsByOriginSkillUseCase`), removing direct KB assignments (`RemoveDirectKnowledgeBaseFromThreadsUseCase`), removing skill-originated sources from threads, cleaning up stale direct thread sources (`CleanupStaleThreadSourcesUseCase`), generating titles via AI inference, and replacing models with user defaults. The add-knowledge-base use case validates org-scoping (the knowledge base must belong to the caller's org) and deduplicates; the remove use case is idempotent when the knowledge base is not assigned. `CleanupStaleThreadSourcesUseCase` computes a 30-day cutoff, asks `ThreadsRepository.findSourceIdsWithOnlyStaleDirectAssignments(olderThan)` for source IDs whose direct (non-skill) thread assignments all point to stale threads, then asks the sources module (`FindUnreferencedSourceIdsUseCase`) to filter down to sources not referenced by any skill, agent, or knowledge base. Deletion is delegated to `DeleteSourcesUseCase`; the result records scanned / unreferenced / deleted / failed counts. `StaleThreadSourcesCleanupTask` runs this cleanup nightly at 2 AM via Nest schedule (staggered with `OrphanedImagesCleanupTask` at 3 AM), skips overlapping executions with an in-memory guard, and logs success or failure counts. Threads carry a `lastActivityAt` timestamp (bumped by `RecordThreadActivityUseCase`, driven by the `ThreadActivityListener` on message-added events) that underpins data retention: `FindExpiredThreadRefsByOrgUseCase` (backed by `ThreadsRepository.findExpiredThreadRefsByOrg`) returns a stably-ordered, paged list of id/owner refs for threads in an org whose `COALESCE(lastActivityAt, createdAt)` predates a cutoff, which the **retention-policies** module consumes to delete aged conversations. Thread title generation uses the language model to auto-summarize conversation content. The `ShareDeletedListener` listens for skill and knowledge base share deletion events: for skill shares it uses `RemoveSkillSourcesFromThreadsUseCase` and `RemoveKnowledgeBaseAssignmentsByOriginSkillUseCase` to clean up assignments that were attached via the revoked skill; for KB shares it uses `RemoveDirectKnowledgeBaseFromThreadsUseCase` to remove direct KB assignments — targeting only users who lost access (resolved via `ShareScopeResolverService.resolveLostAccessUserIds`). The `ThreadMapper` was refactored to use private helper methods (`mapModel`, `mapSourceAssignments`, `mapMcpIntegrationIds`, `mapKnowledgeBases`, `mapMessages`) for readability and to support the new `knowledgeBases` field. The `LocalThreadAssignmentsRepository` service handles all thread assignment persistence operations — source assignments, knowledge base assignments, MCP integrations, and the thread-scoped stale-assignment discovery used by scheduled cleanup — extracted to keep the main `LocalThreadsRepository` focused on core thread CRUD. Cross-module reachability (skill/agent/KB) is handled by the sources module, not here. The HTTP layer exposes `POST /threads/:id/knowledge-bases/:knowledgeBaseId` (add) and `DELETE /threads/:id/knowledge-bases/:knowledgeBaseId` (remove), both returning 204 No Content. MCP integrations are exposed the same way via `ThreadMcpIntegrationsController`: `POST /threads/:id/mcp-integrations/:mcpIntegrationId` (add) and `DELETE /threads/:id/mcp-integrations/:mcpIntegrationId` (remove), both 204. `AddMcpIntegrationToThreadUseCase` validates the integration exists in the caller's org (via the mcp module's `GetMcpIntegrationsByIdsUseCase`, throwing `McpIntegrationNotFoundError` otherwise) and deduplicates; `RemoveMcpIntegrationFromThreadUseCase` is idempotent when the integration is not assigned. The `GetThreadResponseDto` and `GetThreadDtoMapper` include the `knowledgeBases` array (as `KnowledgeBaseSummaryResponseDto`) and the `mcpIntegrations` array (as `McpIntegrationSummaryResponseDto` with `id`, `name`, and marketplace `logoUrl`); integration summaries are resolved in `ThreadsController.findOne` (not in `FindThreadUseCase`, which is on the run-execution hot path). The module integrates with **messages** for conversation content storage, **models** for model selection per thread, **agents** for agent-bound conversations, **sources** for attaching RAG data and deleting stale direct thread sources, **knowledge-bases** for attaching org-scoped knowledge bases to threads, **mcp** for validating MCP integrations before attaching them to threads (via `GetMcpIntegrationsByIdsUseCase`), **runs** which execute within thread context, **shares** indirectly through skill share deletion events, and **IAM** (users, teams) for resolving which users lost access on share revocation.

Domain events are defined in `application/events/`: `ThreadMessageAddedEvent` (emitted when a message is added to a thread, includes userId, orgId, threadId, and current messageCount). The event class exposes a static `EVENT_NAME` string for type-safe listener registration.

Per-thread object-storage cleanup (message-image attachments and generated-image blobs) is prefix-based: thread blobs live under `<orgId>/<threadId>/` and `generated-images/<orgId>/<threadId>/`, so `DeleteThreadUseCase` deletes the thread row first and then best-effort purges those prefixes via the storage module's `PurgeStoragePrefixesUseCase` (a failed delete never costs a surviving thread its blobs; a failed purge only leaks orphans, which the org-level purge sweeps on org deletion). `ThreadsUserDeletionRequestedListener` follows the two-phase deletion contract (`DeferredCleanupEvent`): on `UserDeletionRequestedEvent` it resolves the user's thread ids while the rows still exist and defers a prefix purge (`<orgId>/<threadId>/` and `generated-images/<orgId>/<threadId>/` via the storage module's `PurgeStoragePrefixesUseCase`) that the users module runs only after the row delete succeeds. Lookup failures are logged, not thrown, so deletion is never blocked.

The threads module also manages generated images produced during conversations. The `GeneratedImage` entity represents an AI-generated image linked to a thread, stored via the `GeneratedImagesRepository` port (implemented by `LocalGeneratedImagesRepository`). `SaveGeneratedImageUseCase` persists a generated image record, and `ResolveGeneratedImageUseCase` retrieves an image by ID and returns a pre-signed URL for download. The `GeneratedImagesController` exposes the HTTP endpoint for resolving generated image URLs. Error classes `GeneratedImageNotFoundError` and `GeneratedImageSaveFailedError` handle failure cases for image lookup and persistence respectively.

**Future work:** MCP integrations on threads currently use a bare `@ManyToMany` join table (`thread_mcp_integrations`) without provenance tracking. When user-group-scoped MCP integration access is added, this table should be promoted to a proper entity (like `ThreadSourceAssignmentRecord`) with an `originSkillId` (or similar provenance) column, following the same pattern used for source assignments. This will enable the same cleanup-on-revocation behavior for MCP integrations.
